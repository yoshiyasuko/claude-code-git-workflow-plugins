Conventional Commits形式でコミットを作成し、必要に応じてプッシュ・デプロイまで実行する。変更を論理単位に分割することで、レビューしやすく bisect しやすい履歴を作る。

## モード制御

$ARGUMENTS

| 引数 | 効果 |
|------|------|
| `skip-hooks` | pre-commit / post-push フックをスキップ |
| `skip-push` | プッシュ確認ステップをスキップ |

引数が空の場合は全ステップを実行する（デフォルト動作）。

## 手順

### 1. 変更状況の確認

以下を並列で実行して現在の状態を把握する：
- `git status`
- `git diff`
- `git diff --cached`
- `git log --oneline -10`

### 2. pre-commit フックの実行

**引数に `skip-hooks` が含まれている場合はこのステップをスキップする。**

プロジェクトルートの `.claude/skill-hooks.md` を Read ツールで読み込み、`## commit` セクションの `pre-commit` フックに指定されたスキルがあれば、Skill ツールでそのスキルを実行する。
ファイルが存在しない、セクションがない、または該当フックが定義されていない場合はこのステップをスキップする。

### 3. 論理的な単位への分割

変更内容を分析し、論理的にまとまった単位ごとにコミットを分割する。

1つのコミットに無関係な変更が混ざると、revert や cherry-pick が困難になる。機能追加とバグ修正、リファクタリングと新機能は別コミットにする。

ファイル単位または変更単位で `git add` を使い分ける。`git add -A` や `git add .` はシークレットや不要ファイルを巻き込むリスクがあるため使わず、個別にファイルを指定する。

### 4. コミットメッセージの作成

Conventional Commits (git-cz) 形式で記述する。`git log` で確認した既存スタイルとも整合させる。

```
<type>(<scope>): <subject>

<body>
```

**type**:
- `feat` / `fix` / `docs` / `style` / `refactor` / `perf` / `test` / `build` / `ci` / `chore`

**scope**: 変更対象の範囲（省略可）
**subject**: 変更内容の簡潔な説明。英語・命令形・末尾ピリオドなし。
**body**: 変更の動機や詳細（省略可）。英語で記述。

### 5. コミットの実行

コミットメッセージはHEREDOCで渡す（シェルの特殊文字問題を回避するため）：
```bash
git commit -m "$(cat <<'EOF'
<type>(<scope>): <subject>

<body>
EOF
)"
```

複数コミットが必要な場合は、1つずつステージ→コミットを繰り返す。

### 6. コミット結果の表示

```bash
git log --oneline -10
```

### 7. プッシュの確認

**引数に `skip-push` が含まれている場合はこのステップをスキップする。**

AskUserQuestionツールで確認する：
- 質問文: 「リモートにプッシュしますか？」
- 選択肢: ["はい", "いいえ"]
- 「はい」の場合のみ `git push` を実行

### 8. post-push フックの実行

**引数に `skip-hooks` が含まれている場合はこのステップをスキップする。**

プロジェクトルートの `.claude/skill-hooks.md` を Read ツールで読み込み、`## commit` セクションの `post-push` フックに指定されたスキルがあれば、「説明」列に記載された質問文と選択肢で AskUserQuestion ツールを使って確認し、肯定的な選択肢が選ばれた場合に Skill ツールで指定スキルを実行する。
ファイルが存在しない、セクションがない、または該当フックが定義されていない場合はこのステップをスキップする。

## 守るべきルール

- シークレット情報を含むファイル（.env, credentials等）はコミットしない
- pre-commitフックが失敗した場合は問題を修正して**新しいコミット**を作成する（--amendすると前のコミットを壊すリスクがある）
- `--no-verify` や `--no-gpg-sign` は使用しない（フック失敗は根本原因を修正する）
- 変更がない場合は空コミットを作成しない
